   - HTTP 메서드 기반 라우팅
   - 헤더 기반 라우팅

   - 요청의 속성, 사용자 정보, 인증 상태, 데이터 타입 ...
2. API 통합 및 집중화
   - 분산된 서비스들은 각자 독립적으로 개발, 배포, 운영
   - 단일 인터페이스 : 여러 개의 마이크로서비스 API를 단일 진입점으로 통합, 개발과 통신 편의성 향상
   - 프로토콜 및 데이터 변환


3. 로드 밸런싱 시스템의 요구사항, 부하 분산의 목적, 서비스 인스턴스의 특성 등을 고려하여 결정.
   1. 라운드 로빈 (Round Robin): 라운드 로빈 알고리즘은 요청을 순서대로 서비스 인스턴스에 분배(각 서비스 인스턴스가 동일한 처리 능력을 가정할 때 효과적)
   2. 가중치 기반 (Weighted): 가중치 기반 알고리즘은 서비스 인스턴스에 가중치를 할당하여 요청을 분배(각 서비스 인스턴스는 처리 능력이 다를 수 있으므로, 가중치를 설정하여 더 많은 요청을 처리할 수
      있는 서비스 인스턴스에 더 많은 트래픽을 전달 가능)
   3. 최소 연결 (Least Connection): 최소 연결 알고리즘은 현재 연결 상태를 고려하여 요청을 가장 적은 연결을 가진 서비스 인스턴스에 분배(각 서비스 인스턴스의 부하 상태를 고려하여 트래픽을
      분산하므로, 부하가 균일하게 분배)
   4. 가장 짧은 지연시간 (Least Response Time)
   5. IP 해싱 (IP Hashing)
   6. 랜덤 (Random)
4. 트래픽 관리
   - 요청 제한 (Rate Limiting): 단위 시간당 허용되는 최대 요청 수를 제한하여 트래픽을 제어 가능(ex. 초당 100개의 요청으로 제한할 경우 초과된 요청은 거부되거나 대기열에 추가)
   - 회로 차단기 (Circuit Breaker): 백엔드 서비스의 장애 상황을 감지하고, 일시적으로 해당 서비스와의 통신을 차단함으로써 전체 시스템에 영향을 줄이는 기능
   - 품질 기반 라우팅 (Quality of Service-based Routing)
   - 요청 변환 및 포매팅 (Request Transformation and Formatting): 클라이언트의 요청을 백엔드 서비스가 이해할 수 있는 형식으로 변환하거나, 요청 데이터의 포맷을 일관성 있게
     조정
   - 오류 처리와 재시도 (Error Handling and Retries): 백엔드 서비스에서 발생하는 오류를 처리하고, 재시도 (Fail Back)
5. 보안과 암호화
   - SSL/TLS 사용: API Gateway와 클라이언트 간의 통신을 안전하게 보호하기 위해 SSL (Secure Sockets Layer) 또는 TLS (Transport Layer Security) 프로토콜을 사용. SSL/TLS 인증서를 구입하거나 자체 서명된 인증서를 생성하여 API Gateway에 설치하여 암호화된 연결을 활성화.
   - HTTPS 지원: API Gateway는 HTTPS (HTTP Secure) 프로토콜을 지원. HTTPS는 HTTP 통신을 암호화하여 데이터의 기밀성과 무결성을 보호. 클라이언트는 API Gateway에 대한 모든 요청을 HTTPS로 보내야 하며, API Gateway는 이를 수락하고 처리.
   - 암호화된 페이로드 전송: API Gateway를 통해 전달되는 요청 및 응답의 페이로드를 암호화하여 데이터의 기밀성을 유지. 요청 데이터를 암호화하여 전송하고, API Gateway에서 이를 복호화하여 백엔드 서비스로 전달하는 방식을 사용. 백엔드에서의 응답 데이터도 암호화하여 API Gateway를 통해 클라이언트에게 전송 가능.
   - 암호화된 저장소 사용: API Gateway에서 저장하는 구성 정보, 인증서, 토큰 등의 중요한 데이터는 암호화된 저장소에 저장. 암호화된 저장소를 사용하여 데이터의 기밀성을 보호하고 불법적인 액세스로부터 보호.
   - 암호화 키 관리: 키를 안전하게 저장하고 액세스를 제한함으로써 키의 노출을 방지하고 데이터의 안전성을 보장.
   - 보안 프로토콜 및 알고리즘 사용: 적절한 암호화 알고리즘 및 프로토콜 (예: AES, RSA, HMAC-SHA256)을 선택하여, 암호화 설정을 강화하여 보안 수준을 향상.
   - 접근 제어 및 인증: API Gateway에 접근하는 클라이언트의 신원 확인 및 인증 절차를 구현하여 무단 액세스를 방지. 클라이언트 인증을 위한 다양한 인증 메커니즘을 지원. ex. API 키, 액세스 토큰, 사용자 이름과 비밀번호, OAuth 등을 활용하여 인증을 수행. 유효한 사용자만이 API Gateway에 액세스하고, 암호화된 통신 채널을 통해 데이터를 전송 가능.
   - 보안 로깅: 액세스 시도, 인증 실패, 보안 이벤트, 암호화 키 관리 등. 보안 로그를 기록하여 보안 이슈를 식별하고 조치할 수 있으며, 보안 감사 및 규정 준수를 위한 중요한 요소.
   - 보안 감사 및 모니터링: 액세스 로그, 인증 이벤트, 암호화 관련 이벤트 등을 모니터링하고 분석하여 보안 위협을 탐지하고 대응 필요. 정기적인 보안 감사와 모니터링은 보안 상태를 모니터링하고 보안 정책을 준수 가능.
   - 보안 업데이트 및 패치: API Gateway를 구성하는 소프트웨어 및 하드웨어 구성 요소의 보안 업데이트와 패치를 수행. 이는 취약점이나 보안 이슈를 해결하고, 최신 보안 기능을 활용하여 시스템을 보호하는 데 도움. 보안 패치 및 업데이트를 정기적으로 적용하여 보안 수준을 유지하는 것이 중요.
   - 보안 테스트: 취약성 스캐닝, 펜트레이션 테스트 등. 보안 테스트를 통해 시스템의 취약점을 식별하고 보안 조치를 취해야 데이터 탈취등의 위험으로부터 안전하게 보호.
   - 보안 인시던트 대응(보안 이벤트) 조직의 정보 시스템 또는 중요한 데이터의 기밀성, 무결성 또는 가용성을 위협하는 모든 디지털 침해 또는 물리적 침해  보안 위반 사례, 데이터 노출, 시스템 침해 등의 상황에 대비하여 조치.
   - 보안 교육과 인식: API Gateway 보안 정책, 절차, 최선의 보안 관행 등에 대한 교육을 제공하여 API Gateway 사용자들이 보안에 대한 인식을 높여, 적절한 보안 조치를 통한 보안 위협에 대한 인식을 강화하고 보안 사고에 대한 예방이 가능
   - 외부 보안 감사: 보안 상태와 취약성을 정기적으로 감사하는 외부 보안 감사를 수행. 외부 보안 전문가들이 시스템을 평가하고 취약점을 찾아내며, 보안 위험을 식별하고 보완 조치를 적용. 외부 보안 감사는 독립적인 시각과 통찰력을 제공하여 보안 수준을 향상시킬 수 있는 중요한 절차
   - 규정 준수: 개인정보 보호법, 금융 기관 보안 규정 등. 규정 준수를 통해 법적 요구 사항을 준수하고 사용자들의 개인정보 및 기밀 정보를 안전하게 처리하는 것이 중요.
6. API 모니터링과 로깅
   1. 트래픽 및 요청량 (Traffic and Request Volume)
      초당 요청 수, 요청 유형 및 트래픽 패턴 등을 포함. 트래픽의 증가 또는 감소 파악 및 서버의 부하 예측, 대응 가능.
   2. 응답 시간 (Response Time)
      응답 시간 모니터링은 성능 이슈를 감지하고, 지연이 발생하는 요청을 식별하여 최적화 가능.
   3. 에러 및 예외 (Errors and Exceptions)
      오류 응답의 유형, 발생 빈도, 원인 등. 오류 유형을 식별하고, 서비스의 안정성을 향상시키기 위해 필요한 조치 가능.
   4. 로그 (Logging)
      API 요청의 세부 정보, 요청 헤더, 응답 상태 코드, 오류 메시지 등을 포함. 로그 기록을 통해 디버깅, 오류 추적, 보안 감사 등 다양한 목적으로 활용 가능.
   5. 인증 및 권한 부여 (Authentication and Authorization)
      인증 실패, 접근 거부, 권한 부여 이벤트 등을 포함 인증 및 권한 부여 모니터링은 보안 상태를 파악하고, 잘못된 인증 시도나 권한 위반을 탐지.
   6. 서비스 가용성 (Service Availability)
      API Gateway 자체의 가용성, 연결된 백엔드 서비스의 가용성, 네트워크 연결 등을 포함 서비스 가용성. 모니터링은 장애 발생 여부를 식별하고, 서비스의 신뢰성과 안정성을 보장하기 위해 필요.
   7. 성능 지표 (Performance Metrics)
      응답 시간, 처리량, 대기 시간 등의 성능 지표를 포함. 성능 지표 모니터링은 서비스의 성능 향상을 위해 병목 현상을 파악하고 최적화 가능.
   8. 보안 이벤트 (Security Events)
      악성 요청, DDoS 공격 시도, 인증 실패 등 보안 위협에 관련된 이벤트.
   9. 요청 및 응답 헤더 (Request and Response Headers)
      요청 및 응답 헤더의 값, 크기, 캐싱 여부 등을 포함. 요청 및 응답 헤더 모니터링은 헤더 관련 이슈를 파악하고, 헤더 조작으로 인한 보안 문제를 탐지 가능.
   10. 서비스 경로 및 엔드포인트 (Service Paths and Endpoints)
       각 경로와 엔드포인트의 요청 수, 응답 시간, 성공률 등을 포함. 서비스 경로 및 엔드포인트 모니터링은 특정 경로 또는 엔드포인트에서의 이슈를 파악하고, 서비스 전반의 성능을 개선할 수 있도록 도와줌.
   11. 요청 및 응답 페이로드 (Request and Response Payload)
       요청과 응답의 데이터 크기, 형식, 구조 등을 포함. 페이로드 모니터링은 데이터 무결성, 데이터 유효성 검사, 너무 큰 페이로드로 인한 성능 저하 등을 식별.
   12. 비용 및 요금 청구 (Cost and Billing)
       각 API 호출에 따른 비용, 서비스 사용량, 요금 청구 내역 등을 포함. 비용 및 요금 청구 모니터링은 비용 최적화 및 예산 관리를 위해 필요한 정보를 제공.
   13. 실시간 모니터링 대시보드 (Real-time Monitoring Dashboard)
       시각화하여 제공하는 인터페이스 대시보드는 시스템 상태, 성능, 트래픽 패턴, 오류 등의 정보를 실시간으로 표시하여 관리자가 서비스 상태를 쉽게 파악하고 필요한 조치를 취할 수 있도록 도와줌.
   14. 알림 및 경고 시스템 (Alerting and Notification System)
       이메일, SMS, 푸시 알림 등의 방식으로 관리자나 운영팀에게 경고를 전달하는 기능. 알림 및 경고 시스템은 신속한 대응과 서비스 회복을 돕는 중요한 도구.
   15. 히스토리 및 추적 (History and Tracing)
       각 요청의 시작 시간, 종료 시간, 호출자 정보, 백엔드 서비스 연동 로그 등을 포함 히스토리와 추적 기능은 문제 해결, 디버깅, 오류 추적, 감사 및 규정 준수를 위해 필요.
7. 인증과 인가

8. 캐싱
   - Reverse Proxy 캐싱: API Gateway를 구성하는 프록시 서버가 응답을 캐시하여 다음에 동일한 요청이 들어올 경우 캐시된 응답을 반환 (ex. Nginx, Apache HTTP Server와 같은 웹 서버를 API Gateway로 사용하고, 해당 서버의 캐싱 기능을 활성화하여 캐싱을 구현)
   - 인메모리 캐싱: Redis, Memcached와 같은 인메모리 데이터베이스를 사용하여 캐싱 구현
   - 클라우드 기반 캐싱: Amazon CloudFront, Azure CDN, Google Cloud CDN과 같은 클라우드 기반의 콘텐츠 전송 네트워크 서비스를 사용하여 캐싱을 구현. 서비스는 전 세계에 분산된 캐시 서버를 통해 콘텐츠를 캐시하고, 사용자에게 가장 가까운 서버에서 응답을 제공함으로써 빠른 응답 속도와 효율적인 트래픽 관리를 지원
9. 성능 최적화
   - 캐싱 활용: 자주 요청되는 데이터나 응답을 캐싱하여 빠른 응답을 제공할 수 있습니다. 정적인 데이터나 빈번하게 변경되지 않는 데이터는 캐시에 저장하여 중복 요청을 처리하고 네트워크 대역폭을 절약할 수 있습니다.
   - 로드 밸런싱: API Gateway 앞에 로드 밸런서를 추가하여 서버에 대한 부하를 분산시킬 수 있습니다. 여러 서버에 요청을 분산하고 가용성과 확장성을 향상시킵니다.
   - 비동기 처리: 동기적인 API 호출을 비동기적으로 처리하여 응답 대기 시간을 최소화할 수 있습니다. 비동기 처리는 병렬 처리를 가능하게 하며, 다수의 요청을 동시에 처리할 수 있는 성능 향상을 가져올 수 있습니다.
   - 요청 및 응답 압축: 요청과 응답 데이터를 압축하여 네트워크 대역폭을 절약하고 전송 속도를 향상시킬 수 있습니다. 압축 알고리즘을 사용하여 데이터 크기를 최소화하는 것이 중요합니다.
   - 성능 테스트: API Gateway의 성능을 정기적으로 테스트하여 병목 현상이나 성능 이슈를 식별하고 개선할 수 있습니다. 부하 테스트, 응답 시간 측정, 트래픽 분석 등을 통해 성능을 모니터링하고 최적화 작업을 수행할 수 있습니다.\
   - 최적화된 프로토콜과 포맷: 효율적인 프로토콜 및 데이터 포맷을 선택하여 데이터 전송 속도를 향상시킬 수 있습니다. 예를 들어, HTTP/2 프로토콜은 다중 스트림 및 헤더 압축 기능을 제공하여 성능을 개선할 수 있습니다.
   - 데이터 파이프라인 최적화: API Gateway와 백엔드 서비스 간의 데이터 파이프라인을 최적화하여 데이터 전송과 변환 작업을 최소화할 수 있습니다. 불필요한 데이터 처리를 제거하고 필요한 데이터만 전달함으로써 성능을 향상시킬 수 있습니다.
   - 최적화된 인증 및 권한 부여: 인증 및 권한 부여 과정을 최적
10. API 버전 관리
    - URL 경로 기반 버전 관리 : EX. "/v1/users"와 "/v2/users" 버전을 경로에 명시. API 버전별로 다른 엔드포인트를 유지하고, 각 버전의 변경사항을 독립적으로 관리
    - 헤더 기반 버전 관리:클라이언트가 요청 헤더에 "X-API-Version: 1"과 같은 버전 정보를 전달. URL 경로에 버전을 노출하지 않고도 API 버전 관리.
    - 환경별 버전 관리: 각 환경에 따라 다른 API 버전을 구성하고 관리하여, 개발과 테스트 중에도 프로덕션 환경에서의 안정성을 유지 가능.
    - 백엔드 서비스의 업데이트 관리: 새로운 버전의 백엔드 서비스 배포, 이전 버전의 서비스와의 호환성을 유지하면서 클라이언트로부터의 요청을 적절히 분배 가능.
